# 安卓课程设计：本地 Markdown 轻笔记（开发文档）

本文档综合课程要求、技术选型、架构说明、分阶段计划与**最小可用**的仓库结构，供三人小组本地开发与交付使用。

---

## 1. 项目定位

做一个**有明确主功能、难度适中**的中小型项目：**以本地数据为主的 Markdown 轻笔记**（可命名如「课程笔记库」）。业务主体在 **Web（React 离线包）** 中完成，Android 原生层满足课程对 **Activity / Intent / 控件与菜单 / SQLite** 的硬性要求。

**范围说明**：以**本地开发、完成主流程**为优先，不上完整后端；可选加分项（网络、广播、服务等）视时间从文档第 6 节挑选。

### 1.1 产品能力：仿 Obsidian 的 Markdown 阅读与笔记库（基本功能清单）

以下从**轻量本地笔记库 + Markdown 阅读/编辑**角度列出「仿 Obsidian」应具备或可渐进实现的能力，便于产品范围与课设验收对齐；**加粗**为建议优先实现的**最小闭环**，其余可按里程碑裁剪。

- **库与内容组织**：**单库内多篇笔记**；侧栏或列表展示**笔记列表/标题**；按**更新时间**排序；可选文件夹路径或简单分组（与 Obsidian 的「库根目录下多 .md」对应，可先在 DB 用 `path` 或 `folder` 字段表达）。
- **阅读与编辑**：**打开一篇、查看 Markdown 渲染结果**；**源码编辑**与**预览**至少一种完整、另一种可用简化切换（分屏为加分）；**新建 / 保存 / 删除**笔记。
- **编辑体验（阅读器向）**：基础 **GFM**（表格、任务列表等）渲染；**代码块**高亮可列为阶段目标；长文滚动与焦点在编辑/预览间切换。
- **导航与信息架构**：**从列表进入详情再返回**；可选「**最近打开**」列表（用 DB 或内存记录 `last_opened_at`）；内部链接 `[[其他笔记标题]]` 的解析与跳转可先做**同库标题匹配**再扩展路径语法。
- **发现内容**：**标题或正文关键词搜索**（`LIKE` 或简化的 FTS，时间允许再上单表 FTS5）；**标签**可先做「逗号分隔字段」或独立关联表，图谱类能力为远期。
- **元数据与扩展**：`title`、`updated_at` 必备；**front matter**（YAML）可做解析加分项；**附件/内嵌图片**在离线包形态下可约定放在 `assets` 子目录 + DB 存相对路径，或课设期先不做。
- **外观与可访问性**：**浅色/深色**或跟系统；字号、行宽可在 Web 层实现，原生侧可通过 `Intent` 传主题偏好至壳或只存 SQLite `settings` 表由 H5 读取。
- **数据主权**：**SQLite 本地持久化**；支持将单篇**导出为 .md 文本**或分享（可选）；不做云同步时仍保持「杀进程后数据仍在」的验收点。

> **课设与「重体验」的边界**：以 **SQLite + 离线 Web 主 UI** 能跑通**列表—编辑/预览—落库**为主；全量 Obsidian 插件生态、社区同步、关系图谱不纳入默认范围。

## 2. 与课程要求的对照

| 要求 | 落地方式 | 可验收要点 |
|------|----------|------------|
| 3～4 种控件组合、菜单、事件处理 | **本方案采用离线包为主**：**不以**「多种原生控件搭业务页」为卖点；用原生 `WebView` + 壳上少量控件（如 `Toolbar`、`TextView`/`Button`）与菜单、返回键等满足课程**最低**呈现即可，**主交互在 H5** | 能指出至少三种控件与至少一处菜单/点击事件；业务功能可在 WebView 中演示 |
| 至少 3 个 `Activity` + `Intent` 跳转与传参 | 例如：启动页 → 主界面（壳 + WebView）→ 设置/关于或带 `noteId` 的页 | 三个 `Activity` 类；至少一条链路上有 `putExtra` / 读取 `Extra` |
| 数据存储与访问（SQLite 为高分项） | 在 **Java 层** 使用 `SQLiteOpenHelper` 等实现增删改查，经 **JavascriptInterface** 或等价方式供前端调用 | 杀进程后重进数据仍在；有表结构说明 |

若课程对「纯原生 UI 比例」有额外要求，以任课教师说明为准；本方案在多数院校可接受**混合壳 + 业务 Web** 的说明方式。

## 3. 技术选型

| 层次 | 选型 | 说明 |
|------|------|------|
| 业务 UI | **React** 系列，构建为**离线静态资源**；路由用 **react-router-dom**（推荐 **HashRouter** 以兼容 `file://` 离线包）；样式用 **Sass（SCSS）** 编写，**不采用 Tailwind CSS** | 与团队前端能力匹配；`pnpm` 管理依赖（团队约定使用 pnpm 安装包） |
| 客户端 | **Android 原生 + Java** + `WebView` 加载 `assets` 内 **离线包** | 不引入 Flutter/RN；业务在 Web，原生满足课程点 |
| Markdown | 推荐 **`react-markdown` + `remark-gfm`**；若强调使用 `markdown-it`，可用 **`markdown-it` + 净化策略** 再渲染，避免 XSS 盲区 | 答辩时可说明选库理由 |
| 数据 | **SQLite** 放在 **Android Java 层**，关键 SQL 不在 JS 中拼接 | 满足「数据库存储」高分方向 |
| 后端 | **默认不上**。若需网络加分，再考虑极简 Spring Boot 或仅演示拉取一段文本，以小组精力为准 | 能不上则不上 |

### 3.1 运行形态

- 前端执行 `pnpm build` 得到 `dist`。
- 将 `dist` 放入 `android/app/src/main/assets/dist/`（可脚本同步或按 README 手工拷贝）。
- `WebView` 加载 `file:///android_asset/dist/index.html`；构建时前端需设置**相对资源路径**（如 Vite 的 `base: './'`），避免子路径 404。

### 3.2 与原生通信

- 使用 `WebView.addJavascriptInterface(…, 注入名)`；**注入名**与小组在 H5 里访问的 `window` 对象名一致（见下 3.2.1，例如 `$hybrid`）。
- 能力列表（列表、按 id 读、保存、删除等）通过 **两参数** 的 `invoke` 分发，**不**在端侧为每个能力各挂一个无参/杂参的 `@JavascriptInterface` 方法名到 `window` 上零散暴露，便于扩展示例与联调；具体 JSON 形状见「桥接 API 清单」附录（成稿后冻结）。

#### 3.2.1 `$hybrid` 与 `JSBridge` 的 `invoke` 设计（需一致）

- **`$hybrid`（端侧/客户端）**  
  在 Android 注入给前端的**唯一入口对象**上，统一只暴露**一个**可调方法：`invoke`。端能力（SQLite、Intent、系统能力等）都在客户端组装为**可路由的分发**（例如 `invoke` 内按第一个参数分派到 Java 的私有方法或 Repository），H5 通过 **`$hybrid.invoke(端方法名, 传参)`** 调用。  
  - **第一参数 `端方法名`**：字符串，与协议表（如 `listNotes`、`getNoteById`）一一对应，表示本次要调用的**原生侧逻辑名**。  
  - **第二参数 `传参`**：与该方法配套的入参，一般为 **可序列化为 JSON 的对象/字符串**（如 `{ "id": 1 }` 或空对象）；由前后端在 API 表中对齐，避免在业务组件里写死 `eval` 式拼接。

- **`JSBridge`（H5 侧）**  
  与上**同样设计**：在 `web/src/lib/` 等位置封装一层 **JSBridge**，对业务代码也只暴露**统一的** **`invoke(方法名, 传参)`**（`方法名` 与上文的 `端方法名`、API 表一致；`传参` 为同一套约定）。内部实现为：有真实 WebView 环境则 **`$hybrid.invoke(方法名, 传参)`**（或经一次 JSON .stringify 再交端），无桥（浏览器 `pnpm dev`）则走 **mock**，对调用方**签名不变**。  
  这样 **端对象 `$hybrid` 与 H5 包装 `JSBridge` 的调用面一致**，都落在「**方法名 + 一坨结构化参数**」上，联调、新增能力时只增协议表与分发表，不增加新的 `window.XXX` 形全局函数面。

- **与 `addJavascriptInterface` 的对应**  
  Java 侧为 **一个** 带 `@JavascriptInterface` 的类，对外至少包含 **`void invoke(String methodName, String paramsJson)`**（或你们约定的同义签名）；`methodName` 与第一参数一致，`paramsJson` 为第二参数序列化结果。`WebView.addJavascriptInterface(该实例, "注入名")` 后，H5 使用 **`(window as any).注入名.invoke(...)`** 即可；若注入名需避免特殊字符，可用 `hybrid` 等，但团队文档与代码里**统一**同一字符串。

### 3.3 样式（Sass，不用 Tailwind）

- **技术**：使用 **Sass（.scss 语法）** 维护样式，在 Vite 中由官方或社区插件编译即可（依赖以 `pnpm` 安装，如 `sass`）。
- **明确排除**：**不使用 Tailwind CSS** 及以原子化工具类为主的全局方案，避免与课程/仓库约定不一致。
- **书写约定**（与小组统一）：
  - 不使用 Sass 的 `$` 变量；**主题色、间距等**在样式表里用**原生 CSS 自定义属性**（`--name`）声明，在规则中用 `var(--name)` 引用。
  - 使用 **SCSS 嵌套** 组织选择器，保持可读性。
- **组织方式**：`web/src/styles/global.scss` 管全局与主题；**与页面子组件一一对应** 的局部样式放在 `web/src/styles/components/<与 components 中页面目录相同的键>/`（如 `home/`、`app/`），由对应 `components/` 下 TSX 单独 `import`，**不**在 `pages/` 下放样式。

## 4. 数据与模块（主功能范围）

**最小表结构（示例）**：

- `notes`：`id, title, content_md, updated_at`（可增 `is_deleted` 等扩展字段）。

**主功能**：

- 笔记列表、打开编辑、Markdown 预览/编辑（按时间裁剪，至少其一完整）。
- 数据落库、列表查询在 Java 层实现；React 只负责展示与编辑体验。

**可选（时间允许）**：标签、搜索 `LIKE`、导出为 `.md` 等。

### 4.1 SQLite 在本项目中的角色

- **唯一持久化真源（推荐）**：用户笔记与关键元数据放在 **应用私有目录**下的 SQLite 数据库文件中，H5 **不**直接持有长期权威副本；`WebView` 内展示与编辑通过 **JSBridge → Java Repository → SQLite** 读写，避免只存在 `localStorage` 而与课设「数据库存储」要求不一致的争议。
- **与离线包的关系**：`assets/dist` 中的前端资源**不**存用户笔记；即使用户清浏览器式缓存，只要不动 App 数据，**库内容仍以 SQLite 为准**。

### 4.2 库文件与 `SQLiteOpenHelper` 约定

- **库文件名**：如 `notes.db`（与小组统一即可），位于 `Context.getDatabasePath()` 所指向位置，**无需**人工暴露路径给 H5。
- **基类**：自定义 `*OpenHelper` 继承 `SQLiteOpenHelper`，在 `onCreate` 中执行建表与建索引的 SQL；`onUpgrade` 中按**旧版 version → 新版**做迁移（`ALTER TABLE` 或建临时表再拷贝，按变更复杂度选）。
- **版本号**：`DATABASE_VERSION` 与小组文档一致，任何表结构变更**必须**递增 version 并写清迁移步骤，避免用户升级 App 后闪退或脏数据。

### 4.3 表与字段设计（在「最小表」上扩展时）

- **`notes`（主表，建议）**  
  - `id`：`INTEGER PRIMARY KEY AUTOINCREMENT`。  
  - `title`：`TEXT NOT NULL`（可默认「未命名」）。  
  - `content_md`：`TEXT`（整篇 Markdown 正文）。  
  - `updated_at`：`INTEGER`（Unix 毫秒或秒，**全库排序与「最近」依赖字段**）。  
  - 可选扩展：`is_deleted`（`INTEGER` 0/1 软删）、`path`（逻辑路径）、`raw_front_matter`（或拆表存 YAML，课设可后置）。

- **索引**：至少对**列表页常用条件**加索引，例如 `CREATE INDEX IF NOT EXISTS idx_notes_updated ON notes(updated_at DESC)`；若搜索走 `title`/`content_md` 的 `LIKE '%xxx%'`，**普通索引对中间通配作用有限**，课设可说明局限并预留 FTS 升级接口。

- **可选表 `app_settings`（key-value）**：`key TEXT PRIMARY KEY`, `value TEXT`，存主题、字体档位等**需跨 H5 刷新仍保留**的项；与 1.1 节「主题经 Intent 或 SQLite 传递」一致。

- **与第 1.1 节功能映射示例**：  
  - 列表 / 最近：`ORDER BY updated_at` 或增列 `last_opened_at`；  
  - 搜索：`SELECT ... WHERE title LIKE ? OR content_md LIKE ?`（注意注入防护：只用占位符，**禁止**把用户输入直接拼进 SQL 字符串）；  
  - 标签：可先 `tags TEXT` 存 JSON 或逗号分隔，再考虑规范化到 `note_tags` 多对多表。

### 4.4 Repository 层职责（Java）

- 封装**所有**对 SQLite 的访问：`insert`/`update`/`query`/`delete`（及事务边界），`WebAppBridge` **不**在方法内写散落 SQL 字符串，**统一经 Repository** 调用，便于单测、替换与审阅。  
- **事务**：批量导入、清空回收站等用 `db.beginTransaction()` / `setTransactionSuccessful()` / `endTransaction()`，保证**要么全成要么全败**。  
- **查询结果到 DTO**：用 `Cursor` 逐列读出后组装为 `Note` 等 Java 对象，再序列化为 JSON 字符串经 Bridge 回 H5，避免把 `Cursor` 直接露给 JS。

### 4.5 线程与 `WebView` 回调

- `SQLiteDatabase` 的读写若在 `@JavascriptInterface` 触发，默认可能在 **WebView 桥线程**上执行，易与主线程、`WebView` 更新规范冲突；**建议**：Bridge 内将**重活** `post` 到**后台线程/线程池**执行，结束后若要 `evaluateJavascript` 再切回**主线程**对 `WebView` 回调。在文档/答辩中可写明「**DB 不阻塞 UI 线程、回传在主线程**」的约束。

### 4.6 与 Bridge `invoke` 的协议对应（联调表）

- 在附录冻结前，维护一张 **「方法名 → Repository 方法 → SQL 行为」** 表，例如：`listNotes` 对应 `query` 全表或分页；`getNoteById` 对应主键 `query`；`saveNote` 对应 `insert` 或 `update` 并刷新 `updated_at`；`deleteNote` 对应硬删或软删。  
- **入参/出参**均为 JSON 字符串，字段名与前后端**同一套**命名，减少解析歧义。

### 4.7 安全与数据边界

- 数据库文件在**应用沙箱**内，其他 App 一般不可读；**不**在日志中 `Log` 整篇 `content_md`。  
- 若未来有「从文件系统导入 .md」：在 Java 或 H5 读入后**写入 SQLite** 即完成「入库」，仍以 DB 为展示源。

### 4.8 课设与答辩可准备的 SQLite 材料

- **ER 图或单表+可选表**示意；`onCreate` 里实际执行的**建表 SQL 片段**；一次典型 **`list` + `getById` + `save` 的调用链**从 H5 到 `Cursor` 再返回 JSON 的路径说明；**version 升级**时若只改一列的迁移示例。  
- 验收演示：**强杀 App 后再次进入**，笔记仍在，证明**持久化在 SQLite 而非仅内存**。

## 5. 各端开发要点（简版）

### 5.1 前端（`web/`）

- 目录按页面与桥接划分即可（见第 7 节）；**所有页面级样式以 Sass（SCSS）为主**，不引入 **Tailwind**（见 3.3 节）。
- **`pages/`** 只放置对页面入口组件的**再导出**（或 `key` 等极薄包装）；具体 UI 在 **`components/`** 中**按路由分子目录**（例如路由 **`/`** 下用 `components/home/`，**`/note/...`** 下用 `components/note/`），子组件与样式在 **`styles/components/<同名映射>/`** 中对应维护，**不在 `pages/` 下存放样式**。
- 与原生约定 **协议表中的「方法名 + 入参结构」**（经 `$hybrid.invoke` / `JSBridge.invoke` 两参数形式下发，见 3.2.1），业务 UI 只依赖 `lib/` 的 JSBridge，避免写死 `window` 注入名与序列化细节。

### 5.2 Android（`android/`）

- 至少三个 `Activity`（名称可自定）：如启动、主 WebView 壳、设置/关于等。
- `Intent` 传递如主题、`noteId` 等轻量数据。
- `SQLiteOpenHelper` + 简单 Repository；`@JavascriptInterface` 中注意线程与回到 JS 的线程安全（必要时 `webView.post`）。

#### 5.2.1 如何理解本仓库 Android 端架构（LiteObsidian）

阅读原生代码时，可按「谁负责什么、数据怎么走」来理解当前工程，避免把 WebView 当成普通静态页。

- **包与职责（`com.liteobsidian`）**  
  - **`ui`**：`Activity` 与界面流程。启动页负责首屏与跳转；**`MainActivity` 是混合壳**：布局里嵌 `WebView`，在 `setupWebView` 里配置脚本、本地文件访问与 **`addJavascriptInterface`**；设置页等承担课程要求的控件与 `Intent` 传参。  
  - **`bridge`**：`WebAppBridge` 实现 **`JavascriptInterface`**，是 H5 调用 Java 的**唯一注入点**（注入名与代码中 `JS_INTERFACE_NAME` / `window` 上对象名一致，当前为 **`hybrid`**）。业务上宜在 `invoke(methodName, paramsJson)` 内按方法名分派到数据层，而不是为每个能力新增多个 `@JavascriptInterface` 方法。  
  - **`data`（随功能扩展）**：`SQLiteOpenHelper`、Repository 等应放在 Java 层，由 Bridge 调用，而不是让 JS 直接拼 SQL。

- **从安装包到页面的链路**  
  1. `web/` 执行 `pnpm build` 得到 `dist`。  
  2. 将 `dist` 拷贝到 **`android/app/src/main/assets/dist/`**。  
  3. 打包后，运行时 `WebView.loadUrl("file:///android_asset/dist/index.html")`：**`android_asset` 是系统约定的 URL 前缀**，对应 APK 内的 **`assets/`** 根目录，不是文件系统上的绝对路径。  
  4. 前端构建需使用**相对资源路径**（如 Vite `base: './'`），否则离线包内 JS/CSS 引用容易错误。

- **Bridge 与 WebView 的协作关系**  
  - 注入发生在 **`MainActivity.setupWebView`**：`addJavascriptInterface(new WebAppBridge(this), "hybrid")` 把 Java 对象挂到页面的 **`window.hybrid`**。  
  - H5 通过 **`window.hybrid.invoke(端方法名, JSON 字符串)`** 进入 Java；Java 在 `invoke` 内解析参数并调用 SQLite 等，需要回传结果时再通过 **`WebView.evaluateJavascript`** 或 **`webView.post(...)`** 等与线程模型配合，避免在桥接线程直接操作 UI。

- **建议阅读顺序（本仓库）**  
  1. `AndroidManifest.xml`（入口 Activity、exported、主题）。  
  2. `res/layout/activity_main.xml`（`WebView` 的 id 与壳布局）。  
  3. `MainActivity.java`（`H5_URL`、`setupWebView`、`loadUrl`、返回键与 `WebView` 历史栈）。  
  4. `bridge/WebAppBridge.java`（`@JavascriptInterface` 与后续分发表）。  
  5. 其他 `Activity` 与 `Intent` 传参链路。

按上述层次理解后：**原生负责进程内持久化、系统能力与壳导航；H5 负责笔记编辑与展示；两者契约集中在 Bridge 的方法名与 JSON 参数上。**

### 5.3 后端

## 7. 仓库与目录结构（推荐）

**单仓根目录下仅两个主工程**，不强制 monorepo 工具链，降低协作成本。

```text
<仓库根>/
├── README.md                    # 本地运行与打包 dist 的说明
├── web/                         # React，pnpm
│   ├── public/
│   ├── src/
│   │   ├── main.tsx
│   │   ├── App.tsx
│   │   ├── styles/
│   │   │   ├── global.scss
│   │   │   └── components/        # 与 components/ 子目录按路由映射：如 app/、home/、note/
│   │   ├── pages/               # 仅 re-export 页面入口，如 pages/HomePage.tsx
│   │   ├── components/         # 按路由划分子目录，放实际组件（如 app/、home/、note/）
│   │   ├── lib/                 # 桥接封装、小工具
│   │   └── types/
│   ├── index.html
│   ├── package.json
│   ├── pnpm-lock.yaml
│   └── vite.config.ts            # 或其他构建；注意 base: './'
│
├── scripts/                     # 可选：将 web/dist 同步到 android assets
│   └── copy-web-to-android.mjs
│
└── android/                     # Android Studio，Java
    └── app/
        ├── build.gradle
        └── src/main/
            ├── java/<包名>/
            │   ├── ui/            # SplashActivity, MainActivity, SettingsActivity 等
            │   ├── data/         # NoteDbHelper, NoteRepository
            │   └── bridge/       # WebAppBridge（JavascriptInterface）
            ├── assets/
            │   └── dist/         # 由 pnpm build 产物拷贝至此
            ├── res/
            └── AndroidManifest.xml
```

**分工建议**：两名前端以 `web/` 为主；一名 Java 以 `android/` 与 SQLite、桥接为主；熟悉离线包与集成的同学负责构建路径与联调。

## 9. 启动与本地调试

### 9.1 环境准备

- **Node.js**：与当前 LTS 兼容即可；包管理用 **pnpm**（在 `web/` 下执行 `pnpm install`）。
- **Android 开发**：安装 **Android Studio**（含 Android SDK、至少一个**模拟器**或真机 + USB 调试）。本项目壳为 **Java**，Gradle 与 JDK 版本以 `android` 工程与 Android Studio 提示为准。

### 9.2 只调前端（最快）

在仓库中进入 `web/`：

```bash
cd web
pnpm install
pnpm dev
```

- 用浏览器打开终端里给出的本地地址（如 `http://localhost:5173`），可专注改界面、路由、Markdown 等。
- **与原生桥接**在浏览器里没有真实 `window` 上的**注入端对象**（如 `$hybrid`）时，可在 `lib/` 的 **JSBridge** 里做 **判断 + 假数据**（有桥则 `invoke` 到端，无则同签名走 mock），这样不必每次改 UI 都打包进 APK（见 3.2.1）。

### 9.3 与 App 真机/模拟器联调（完整路径）

1. 构建静态资源并放到 Android 可读位置：
   - 在 `web/` 执行 `pnpm build`，得到 `web/dist`。
   - 将 `dist` **整目录**复制到 `android/app/src/main/assets/dist/`（若不存在则先建 `assets/dist`；内容应为 `index.html` 与 `assets/` 等，具体以 Vite 输出为准）。
2. 用 **Android Studio** 打开**工程根下的 `android/`**（或你们 Monorepo 中存放 Gradle 工程的那一层），等待 Gradle 同步完成。
3. 选择运行设备（模拟器或已开启 USB 调试的真机），点击 **Run** 安装并启动 App。

之后每次**改了前端并要在壳里看效果**：重复 `pnpm build` + 覆盖拷贝 `dist` + 在 Android Studio 中再次 **Run**（或仅重启 Activity，以实际改动为准；资源变了一般需重装或清进程后再进）。

### 9.4 调试手段（常用）

- **看前端报错 / 打日志（壳内）**：在 `WebView` 上（API 19+ 常用）可开启 `WebView.setWebContentsDebuggingEnabled(true)`（仅 **debug 构建**建议开启，正式包可关闭）。在电脑 Chrome 打开 `chrome://inspect`，选中对应 **WebView** 即可用开发者工具看 Console、网络等。
- **看原生与 SQLite**：在 Android Studio 用 **Logcat** 过滤你的包名或 TAG；断点可打在 `Activity`、`Repository`、`@JavascriptInterface` 方法上。
- **仅调 Java/布局**：不依赖 H5 变更时，可直接 Android Studio 运行、改 `Activity` 与 `res/layout`，与 Web 解耦。

### 9.5 与「日常开发习惯」的对应

| 场景 | 建议做法 |
|------|----------|
| 改 SCSS、页面结构、Markdown 展示 | 优先 `pnpm dev` + 浏览器，必要时用 mock 桥接 |
| 调列表数据、存库、Intent | 在 Android Studio 里跑，配合 Logcat/断点 |
| 调 H5 与 Java 的接口契约 | 双方对齐 JSON 后，可先在浏览器用 mock 再在设备上验证 |

### 9.6 若暂时还没有 `android/` 工程

- 在 Android Studio 用 **Empty Activity** 新建 Java 工程，包名、最低 SDK 按课程与组长约定设置；将本文档中 `WebView` + `assets/dist` 与多 `Activity` 的约定落到该工程即可。先把「能加载本地 `index.html`」跑通，再叠 SQLite 与桥接。
